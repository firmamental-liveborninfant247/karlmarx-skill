# General Problem Router

## Purpose
Decide whether the package should be used strongly, lightly, or not at all.

---

## Step 1: Identify the dominant problem type

Choose one or more:

1. **Symptom problem**
   The user sees repeated events but not the structure.

2. **Strategy problem**
   The user wants leverage, not just description.

3. **Narrative problem**
   A dominant framing may be misleading.

4. **Historical problem**
   The present arrangement looks natural or timeless.

5. **System problem**
   Interacting parts create outcomes no one actor intends.

6. **Intervention problem**
   The user needs action and testable next steps.

---

## Step 2: Score applicability

### Strong
Use strongly if at least two are true:
- institutions or organizations are central
- incentives or dependency matter
- local fixes are failing
- power or control asymmetries matter
- history/path dependence matters
- framing or legitimation matters

### Medium
Use selectively if:
- the issue has both structural and individual components
- the framework can reveal constraints but not fully explain behavior

### Weak
Do not force if:
- the issue is mostly technical, formal, or clinical
- the answer mainly depends on narrow domain expertise
- structural analysis would add only decorative depth

---

## Step 3: Select foundational moves

Before selecting higher-level engines, select the basic analytical corrections the answer needs.

Recommended defaults for many structural questions:
- relation and totality
- movement and rest
- essence and appearance
- practice as criterion

Add others when needed:
- development and change
- universality and particularity
- cause and effect
- necessity and contingency
- quantity and quality
- negation and transformation
- content and form
- possibility and actuality
- freedom and necessity
- abstraction and concretion
- mediation and levels

If the case is very specific or operational, also activate:
- concrete analysis of concrete conditions

---

## Step 4: Select engines

| Problem signal | Engine |
|---|---|
| repeated crisis or trade-off | contradiction analysis |
| local fixes fail | totality analysis |
| “this is just how things are” | historicization |
| official language feels off | ideology critique |
| user wants action | praxis loop |
| theory is getting too generic | concrete analysis of concrete conditions |

Use only the engines that improve the answer.

---

## Step 5: Compare rival frameworks

Ask which framework should dominate:
- economics
- psychology
- management
- law/policy
- engineering

If a rival framework dominates, either:
- fuse it with this package, or
- step back and use Marxist analysis only as a secondary lens.

---

## Step 6: Output discipline

Every strong-use answer should contain:
1. surface description
2. structural diagnosis
3. principal contradiction or loop
4. hidden framing if relevant
5. historical formation if relevant
6. leverage point
7. certainty labels

If the problem is operational or case-heavy, also include:
8. the concrete conditions that make this case different from the template

---

## Fail-safe line
If applicability is weak, say so explicitly:
> “A structural reading adds a little context here, but the main answer actually depends on X.”
